Red Hat Enterprise Linux (RHEL) サブスクリプション利用規約の解説：サービスプロバイダー向け
I. エグゼクティブサマリー
 * 主要な調査結果： 標準的なRed Hat Enterprise Linux (RHEL) サブスクリプションは、ソフトウェアの使用権、アップデート、パッチ、および内部利用のためのサポートへのアクセスを提供しますが、第三者への再配布や商用サービスの提供は明示的に禁止されています 。RHELを基盤としたサービスを顧客に提供する事業者は、Red Hat認定クラウド&サービスプロバイダー（CCSP）プログラム  に参加する必要があります。これは、そのようなサービス提供形態をRed Hatが公式に認める唯一の方法です 。全てのサブスクリプション利用は、Red Hatエンタープライズ契約（EA） および関連する製品付属文書やエンドユーザーライセンス契約（EULA） に準拠する必要があります。導入規模（ソケット数やインスタンス数など）に基づいたサブスクリプションのコンプライアンス維持は極めて重要です 。
 * 主要な推奨事項： RHELベースのサービス提供を計画している事業者は、標準サブスクリプションではなく、CCSPプログラムへの参加を必須と考えるべきです。事業計画においては、CCSPのライセンスモデルとサポートレベルを慎重に評価することが不可欠です。
 * 免責事項（重要）： 本レポートは、公開されている情報に基づき、情報提供のみを目的として作成された要約です。法的な助言を構成するものではありません。最終的な判断は、Red Hatエンタープライズ契約、関連する付属文書、EULA、およびCCSPプログラムの公式な全文を確認し、必要に応じてRed Hatまたは法的な専門家にご相談の上で行う必要があります。
II. RHELサブスクリプションの基本
 * A. RHELサブスクリプションとは何か？
   * RHELサブスクリプションは、単なるソフトウェアライセンスではなく、定義された期間にわたって一連のサービスと価値へのアクセス権を提供するものです 。これには、テスト・検証済みのエンタープライズ向けLinux OSソフトウェア本体  に加え、継続的なアップデート、バグ修正、セキュリティパッチ（Errata）の提供 、テクニカルサポート 、ソフトウェア管理ツール「Red Hat Network（RHN）」へのアクセス 、関連文書へのアクセス 、そしてハードウェアやソフトウェアの認定情報  などが含まれます。
   * サブスクリプション料金は、Red Hatによる継続的な製品開発、アップストリームのオープンソースプロジェクトへの貢献、そして広範なハードウェア・ソフトウェア・クラウドプロバイダーとのパートナーシップエコシステムの維持・統合に使われます 。これにより、RHELの信頼性と互換性が確保されています。
   * このサブスクリプションモデルは、従来の買い切り型ライセンスとは異なり、Red Hatと顧客との間のサービスベースの関係性を表しています。Red Hatは継続的な価値提供を通じてサブスクリプション更新を促し、顧客は常に最新の機能改善やセキュリティ対策、サポートを享受できるという、双方にとってメリットのある仕組みと言えます。
 * B. 標準的な権利（通常得られるもの）：
   * ソフトウェアアクセス： RHELのバイナリコードとソースコードの両方をダウンロードし、インストールする権利が付与されます 。一つのサブスクリプションで、複数のバージョンのRHELを利用可能です 。サブスクリプション期間中にリリースされる最新バージョンへのアップグレードも可能です 。
   * アップデートとメンテナンス： RHNや、クラウド環境によってはRHUI（Red Hat Update Infrastructure） などを通じて、機能強化、バグ修正、セキュリティパッチ（Errata）、新しいハードウェアサポートを含むアップデートを継続的に受け取ることができます 。
   * サポートアクセス： Red Hatのナレッジベースへのアクセス  に加え、購入したサポートレベルに応じて、ウェブまたは電話によるテクニカルサポートが提供されます 。
   * 管理ツール： ソフトウェア配信やアップデート管理のためのRed Hat Network（RHN） や、環境分析と問題修正を支援するRed Hat Insights  へのアクセスが含まれる場合があります。一部の構成では、Smart Managementアドオンが必要になるか、バンドルされていることがあります 。
   * ライフサイクルサポート： 各メジャーバージョンに対して、通常10年間（標準7年＋延長3年） のサポートが提供され、安定した運用と計画的なシステム更新を可能にします 。
   * 一つのサブスクリプションで複数のRHELバージョンを利用できる柔軟性  は、運用上の大きな利点です。ライセンス体系の変更に縛られることなく、組織自身のスケジュールやリスク許容度に基づいてアップグレード計画を立てることが可能になります 。これは、特定のバージョンに固定されたライセンスモデルと比較して、より自由度の高い運用を実現します。
 * C. 背景にある考え方と影響：
   * サブスクリプション料金がアップストリームプロジェクトへの貢献やエコシステム認定に使われる  という事実は、Red Hatとオープンソースコミュニティ、およびハードウェア・ソフトウェアベンダーとの共生関係を示しています。この関係性が、RHELが「エンタープライズ対応」であることの基盤を形成しており、多くの企業に採用される理由の一つとなっています 。
III. 利用権、制限、および義務
 * A. 許可される利用範囲（標準サブスクリプション）：
   * 標準サブスクリプションは、原則として購入した法人または個人による内部利用を目的としています 。
   * 利用は、サブスクリプションを購入した特定のユニット（例：物理システム、ソケットペア、仮想インスタンス）に関連付けられます 。ライセンスされたユニット数を超えて使用することはできません。
   * ライセンスされたユニット上でソフトウェアを実行し、アップデートやサポートを受けることが許可されます。
   * 自社の内部インフラとしての利用、自社アプリケーションの実行基盤としての利用は許可されます。また、RHELコンポーネントが主要な課金対象サービスではなく、かつ再配布されない限りにおいて、自身のウェブサイトやSoftware-as-a-Service（SaaS）を提供するために内部的に使用することも、特定の条件下では許容される可能性があります 。ただし、この点については慎重な確認が必要です。
 * B. 禁止事項と主要な制限：
   * 再配布・再販の禁止： 標準契約の下では、ソフトウェアやサブスクリプションサービスを複製して第三者に配布したり、再販したりすることは明確に禁止されています 。これは、サービスプロバイダーとしての事業モデルを検討する上で最も重要な制限事項です。
   * 第三者向けホスティング・マネージドサービスの禁止： 標準サブスクリプションを使用して、第三者のためにホスティングサービス、マネージドサービス、あるいはその他のサービスを提供することは、一般的に契約の重大な違反とみなされます 。
   * 回避行為の禁止： 料金の支払いを回避したり、ユニット数の制限を潜脱したりする目的で、複数のアカウントを作成したり、使用状況を操作したりすることは禁止されています 。有償サブスクリプションを通じて得た個人アカウントのアクセス権やバイナリを他者と共有したり、権利のないシステムで使用したりすることも禁じられています 。
   * 改変に関する制限： RHELはオープンソースソフトウェアをベースとしていますが、Red Hatが提供していないパッチを適用したり、サポートされていない改変を行ったりした場合、サブスクリプションに含まれるサポートサービスが無効になる可能性があります 。コードを改変する基本的な権利はGPL（General Public License）によって保証されていますが 、その権利を行使することが、有償のサポート契約の条件に影響を与える場合がある点に注意が必要です。
 * C. コンプライアンス要件と監査：
   * 正確なサブスクリプション数： 顧客は、実際に導入または使用している総ユニット数と同数のサブスクリプションを購入し、維持する必要があります 。この要件は、特定の製品ファミリー全体（例：全てのRHEL Serverインスタンス）に適用されます 。
   * 報告義務： 使用量が購入したユニット数を超過した場合、顧客は速やかに超過分とその使用開始日をRed Hatまたはビジネスパートナーに報告し、追加のサブスクリプション料金を支払う必要があります 。
   * 監査権： Red Hatは、契約条件の遵守状況を確認するため、顧客の施設や記録を検査・レビューする権利（または自己評価を要求する権利）を留保しています 。監査権は、契約期間中および契約終了後1年間有効な場合があります 。
   * 不遵守の場合の結果： 契約違反が発見された場合、顧客は通常、報告不足分の使用量に対する料金を、場合によっては導入時点まで遡って支払うよう要求されます 。コンプライアンス違反が繰り返されると、「信頼に基づくシステム（honor system）」としての運用が難しくなり、より厳格な監査や管理プログラムの対象となる可能性があります 。
   * 標準契約における再配布および第三者向けサービスの厳格な禁止  は、サービス提供を目指す事業者にとって、CCSPプログラムの利用が不可欠であること直接的に示しています。これは曖昧な条項ではなく、明確な障壁です。
   * コンプライアンスが「信頼に基づくシステム」 に依存する一方で、Red Hatが監査権  を保持しているという事実は、顧客側での正確な導入状況の追跡が不可欠であることを意味します。監査で不遵守が発覚した場合の遡及支払いリスク  を回避するためには、事前の内部管理体制の構築が重要となります。
   * GPLに基づくコード改変の権利  と、サブスクリプションサービスの契約条件  は区別して理解する必要があります。オープンソースコードを自由に改変することは可能ですが、Red Hatがサポートする枠組みから外れた変更を行うと、有償で得られる価値（サポート、認定など）を失う可能性があります。利用者は、コードの自由度を最大限に追求するか、商用サポートの適格性を維持するかの選択を迫られる場面があり得ます。
IV. サブスクリプションモデルと種類
 * A. 一般的なRHELサブスクリプションタイプの概要：
   * RHEL Server： 物理、仮想、クラウド環境でのサーバー展開向けの標準製品です 。多くの他の製品の基盤となります。
   * RHEL for Virtual Data Centers (VDC)： 高密度仮想化環境向けに設計されており、通常、ソケットペア単位でライセンスされたハイパーバイザーホスト上で無制限のRHELゲストを実行できます 。特定のゲスト密度を超えると、コスト効率が高くなる可能性があります 。
   * RHEL Workstation / Desktop： エンドユーザーコンピューティング環境向けで、通常、サーバー版とは異なる条件や低価格が設定されています 。原則としてサーバー用途には使用できません 。
   * RHEL Developer Suite / Subscription： 個々の開発者向けの低コストまたは無料のオプションが多く、本番環境での使用は想定されていません 。
   * 特殊なバリエーション： RHEL for SAP Applications / HANA 、RHEL for HPC (Head Node, Compute Node) 、RHEL for Real Time  などがあります。これらは特定のユースケースに合わせて調整された条件、サポート体制、ライセンスメトリックを持つ可能性があります。
   * アカデミックサブスクリプション： 教育機関向けの特別な条件と価格設定が存在します 。
 * B. ライセンスメトリック（課金単位）の説明：
   * 物理環境への導入： 通常、物理サーバーに搭載されているCPUソケットのペア数に基づいてライセンスされます 。標準的なRHEL Serverサブスクリプション1つで、最大2ソケット（1ソケットペア）搭載の物理サーバー1台分をカバーすることが一般的です 。
   * 仮想環境への導入（低～中密度）： RHELを実行している仮想インスタンスのペア数に基づいてライセンスされる場合があります 。標準的なRHEL Serverサブスクリプション1つで、仮想インスタンス2台分をカバーすることがあります 。
   * 仮想環境への導入（高密度）： 多くの場合、ホストサーバーの物理ソケットペア数に基づいてライセンスされ、そのホスト上で無制限のゲストOS（RHEL）を実行できるモデル（例：RHEL for VDC）が適用されます 。
   * スタッキング（積み上げ）： 1つのサブスクリプションがカバーする数を超えるソケットや仮想インスタンスを持つシステムには、複数のサブスクリプションを「積み上げて」適用することができます 。例えば、4ソケットサーバーには、2ソケット対応のサブスクリプションが2つ必要になる場合があります 。
   * 柔軟性・転用： 一部のサブスクリプションでは、物理（ソケットペア）ライセンスと仮想（インスタンスペア）ライセンスの間で権利を転用できる場合があります 。これにより、インフラストラクチャが物理から仮想へ移行した場合などに、サブスクリプションを調整できます。これは通常、RHEL Serverとそのアドオンに適用されます 。
   * クラウドでのライセンス： クラウドプロバイダー経由での提供（多くはCCSPパートナー経由）では、異なるメトリック（例：vCPU数に基づく時間課金や月額課金 ）が用いられることがあります 。Red Hat Cloud Accessプログラムを利用すると、既存のサブスクリプションを認定クラウドプロバイダー環境に持ち込むことも可能です 。

 * D. 背景にある考え方と影響：
   * ソケットペア、インスタンスペア、VDCといった複数のライセンスメトリックが存在することは、物理、仮想化、高密度仮想化といった多様なITインフラの実情にRed Hatが対応していることを示しています。さらに、物理と仮想の間での転用可能性  は、インフラ移行期にある顧客にとって、サブスクリプション投資の柔軟性を高める要素となります。
   * 特定の密度（例：ソケットペアあたり5ゲスト超 ）を超えると無制限ゲストを提供するVDCモデルは、仮想化を計画する顧客にとって明確な経済的判断基準を生み出します。VM密度がある閾値に達した場合、VDCライセンスを持つ少数の強力なホストに集約することが、VMペアごとに課金されるよりも経済的になるため、効率的な仮想化ホストの利用を促すインセンティブとして機能します。
   * SAP、HPC、Real Timeといった特殊なRHELバリアント  の存在は、特定の市場セグメントをターゲットにした戦略を示唆しています。これらは、関連ベンダーとの共同開発や、特定のチューニング、特別なサポートコミットメントを含む可能性が高く、それが異なる条件や価格設定に反映されていると考えられます。
V. RHELを利用したサービス提供：CCSPプログラム
 * A. 標準サブスクリプションによる第三者向けサービスの制限（再確認と強調）：
   * 繰り返しになりますが、標準のRHELサブスクリプションはエンドユーザー自身の内部利用を意図しており、サービスプロバイダー事業の中核となる活動、すなわち再販、第三者向けホスティング、RHELベースのマネージドサービスの提供などは明確に禁止されています 。これらの目的で標準サブスクリプションを使用することは、エンタープライズ契約（EA）の違反となります 。
 * B. 認定クラウド&サービスプロバイダー（CCSP）プログラムの紹介：
   * CCSPは、パートナー（サービス提供を目指す事業者を含む）がRed Hat製品（RHEL、OpenShift、Middlewareなど）を基盤として、自身のエンドカスタマーにサービスを提供することを可能にするRed Hatの公式プログラムです 。
   * Red Hat製品を用いたクラウドサービスやホステッドサービスを提供する、唯一の認定された方法です 。
   * このプログラムは、以前の認定クラウドプロバイダー（CCP）プログラムを拡張したもので、より広範なパートナー（マネージドサービスプロバイダー、システムインテグレーター、通信事業者、ホスティング事業者、VARなど）とハイブリッドクラウドシナリオを包含するようになりました 。
   * CCSPに参加することで、パートナーはホステッドサービスやマネージドサービスを提供し、収益と利益性を向上させる機会を得ることができます 。
 * C. CCSPの仕組み（サービスプロバイダー向け）：
   * ライセンスモデル： 多くの場合、従量課金制または消費ベースのモデルを採用しています。プロバイダーは使用量（例：月次）をRed Hatに報告し、それに応じて料金を支払います 。これは、エンドユーザー向けの一般的な前払い期間契約とは異なり、プロバイダーがコストを収益と連動させることを可能にします。
   * 導入の柔軟性： プロバイダー自身のインフラ上だけでなく、主要なパブリッククラウド（AWS、Azure、GCPなど）上でも利用できる場合があり、マルチクラウド戦略をサポートします 。
   * プロバイダーの義務： CCSPパートナーには特定の要件があります 。
     * 検証・認定プロセスの完了。
     * 技術的な専門知識の維持（トレーニングを受けたスタッフの確保）。
     * スケーラブルで安全なインフラストラクチャの提供。
     * 使用量を計測・請求・報告するシステムの導入・運用。
     * エンドカスタマーがRed Hatのサブスクリプション条件に同意することの保証（契約条件のフローダウン）。
   * サポートモデル： モデルによって異なります。パートナーが顧客への一次サポートを提供するモデル（例：「Partner Diagnostic support model」）もあれば、Red Hatがより直接的に関与するモデルもあります。パッチ適用やアップデートはプロバイダーが管理する場合があります 。
   * パートナー/ディストリビューター： CCSPプログラムへの参加には、特定のディストリビューターを経由する必要がある場合があります 。
   * CCSPは単なるライセンス供与ではなく、相互の義務を伴うパートナーシッププログラムです 。Red Hatはサービス提供のためのソフトウェア利用権を提供し、パートナーは技術標準、報告義務、エンドカスタマーの契約遵守保証などを約束します。これは、標準サブスクリプションの購入よりも参入障壁が高く、継続的な運用コミットメントが必要であることを意味します。
   * CCSPパートナーがエンドカスタマーにRed Hatの利用規約への同意を求める義務  は、重要な法的メカニズムです。これにより、ソフトウェアが仲介者を通じて提供される場合でも、Red Hatの契約条件が最終的なソフトウェア利用者にまで及び、ライセンスコンプライアンスが連鎖的に確保されます。これはRed Hatの知的財産とライセンス条件を保護する上で不可欠です。
   * CCSPで一般的な従量課金モデル  は、サービスプロバイダーのビジネスモデルに合わせて設計されています。顧客の消費量や収益に応じてコストが変動するため、期間契約を一括購入する場合と比較して、初期投資リスクを低減できます。
VI. サポートレベルの理解
 * A. サポート階層と範囲：
   * 一般的な階層： 本番環境向けサポートには、通常、StandardとPremiumのオプションがあります 。また、Red Hatからの直接支援は含まれないものの、ナレッジベースやアップデートへのアクセスを提供するSelf-Support  や、開発者向けのDeveloper Support  も存在する場合があります。
   * 主な違い：
     * サポート時間： Standardは通常、標準営業時間内のみですが、Premiumは多くの場合、クリティカルな問題（Severity 1 & 2）に対して24時間365日のサポートを提供します 。
     * 応答時間： Premiumは一般的に、特に重大度の高い問題に対して、より迅速な初期応答時間を提供します 。
     * ケース数： 本番サポート階層（Standard, Premium）では、通常、サポートケースの数に制限はありません 。
     * サポートチャネル： 有償サポート階層では、通常、ウェブおよび電話でのサポートが提供されます 。
   * 重大度レベル： サポートリクエストは、問題の影響度に基づいて優先順位付けされ（例：Severity 1（本番環境停止）からSeverity 4（影響の少ない問題）まで）、応答時間の目標値に影響します 。
 * B. 表：RHELサポートレベルの比較（例）
| 特徴 | Self-Support  | Standard  | Premium  |
|---|---|---|---|
| 提供時間 | N/A | 標準営業時間 | 標準営業時間（Sev 1/2は24x7） |
| サポートチャネル | ナレッジベース  | Web & 電話 | Web & 電話 |
| ケース数制限 | N/A | 無制限 | 無制限 |
| 応答時間 (Sev 1) | N/A | 1営業 時間（初期） | 1時間（初期） |
| 応答時間 (Sev 2) | N/A | 4営業 時間（初期） | 2時間（初期） |
| 応答時間 (Sev 3) | N/A | 1営業日（初期） | 4営業 時間（初期） |
| 応答時間 (Sev 4) | N/A | 2営業日（初期） | 8営業 時間（初期） |
| 対象ユーザー | テスト/開発 | 本番環境 | ミッションクリティカルな本番環境 |
 * C. サポートに関する主な考慮事項と制限：
   * パッチ適用： Red Hatから直接提供されていないパッチやアップデートを適用すると、サポート契約が無効になる可能性があります 。公式なRed Hatチャネル（RHN, Satellite, RHUIなど）を使用することの重要性がここでも示唆されます。
   * 範囲： サポートは通常、RHEL OS自体と、統合されたコンポーネントを対象とします。サードパーティ製アプリケーション、カスタムコード、サポートされていないハードウェアや構成に起因する問題は、Red Hatのサポート義務の範囲外となる可能性があります 。
   * CCSPサポート： 前述の通り、CCSPプログラム下のサポートモデルは異なる場合があり、パートナーが一次/二次サポートを担当する必要があるかもしれません 。プログラム参加時にこの点を確認することが重要です。
   * StandardとPremiumサポートの選択  は、コストとリスク軽減の直接的なトレードオフです。Premiumは迅速な応答と24時間365日のカバレッジを提供し、高可用性サービスを提供するプロバイダーにとっては不可欠ですが、価格は高くなります。一方、重要度の低いワークロードや、プロバイダー自身が高い内部対応能力を持つ場合は、Standardで十分な場合もあります。この選択は、提供するサービスの重要度と顧客に約束するSLAに基づいて行われるべきです。
   * Red Hat提供外のパッチに関する制限  は、サブスクリプションの価値提案、すなわち単なるソフトウェアではなく検証済みスタックを提供している点を強調しています。外部パッチを使用することは、その検証を破棄し、サポートの責任をRed Hatから逸脱させる行為とみなされます。
VII. 準拠すべき法的枠組み
 * A. Red Hatエンタープライズ契約（EA）の役割：
   * EAは、全てのRed Hat製品およびサービスの購入と使用を規定する基本契約です 。顧客は購入時にこのEAに同意する必要があります 。
   * EAには、一般条件、支払条件、保証、責任制限、コンプライアンス/監査条項、機密保持などが含まれます 。
   * 地域別のバージョン（例：米国、EMEA、APAC、日本）が存在します 。サードパーティのマーケットプレイス経由で購入した場合、特定の地域（例：米国またはEMEA）のEAが適用されることがあります 。
   * 政府機関向けには、特別な補足契約が存在する場合があります 。
 * B. 製品付属文書とエンドユーザーライセンス契約（EULA）：
   * EAは、特定のソフトウェア製品（RHELなど）に関連する条件を含む、個別の製品付属文書やEULAと組み合わせて適用されます 。
   * RHEL自体は通常GPLv2に基づくライセンスで提供されますが、サブスクリプションサービスには追加の商用条件が付加されます 。EULAは、サブスクリプションに関連する権利と制限を明確にします。
   * クラウドサービスでの利用には、サービスプロバイダーが提示する、Red Hatからフローダウンされた特定のEULAや追加条項が適用される場合があります 。例えば、 は富士通の条件がRed HatのクラウドEULAを参照している例であり、 はサービスプロバイダー経由で購入したエンドユーザーとRed Hat間のEULAの一例です。
   * Red Hatサブスクリプションに含まれるサードパーティ製ソフトウェアには、それぞれ独自のEULAが適用されます 。
   * EA、付属文書、EULAという階層的な法的構造  は、特定の製品に関する権利と義務を完全に理解するためには、顧客が複数の文書を確認する必要があることを意味します。主要なEAだけに依存するのは不十分です。
   * EAが「オープンソースライセンスの条件下でのソフトウェアコードに対するあなたの権利を制限することを意図したものではない」 と明記されている点は、重要な明確化です。これにより、基盤となるオープンソースの権利と、サブスクリプションサービスの商用条件が法的に分離されます。GPLを尊重しつつも、有償サービス（サポート、アップデート、認定など）に関するルール（サポート条件  や再配布禁止  など）を強制することが可能になります。
VIII. サービスプロバイダーを目指す上での主要な考慮事項
 * A. CCSP参加は必須条件： 計画しているサービス提供事業モデルでは、CCSPプログラムへの参加が必須です 。標準サブスクリプションを使用しようとすると、ライセンス違反となり契約不履行につながります 。
 * B. CCSPの義務を理解する： CCSPは単に料金を支払うだけでなく、技術、運用、報告、場合によってはサポートに関する要件を満たす必要があるパートナーシッププログラムです 。これらの義務を事業計画に織り込む必要があります。
 * C. コストを正確にモデル化する： 計画するサービスアーキテクチャに対して、RHELのライセンスメトリック（ソケット、インスタンス、VDC、CCSPの消費モデル）を慎重に評価し、コストを正確に予測することが重要です 。
 * D. サポートレベルを整合させる： 自社の顧客に提供する予定のSLA（サービス品質保証）に合わせて、Red Hatのサポート階層（Standard、Premium、またはCCSP固有のモデル）を選択する必要があります 。
 * E. コンプライアンスは最重要： RHELの導入と使用状況を追跡するための堅牢な内部プロセスを実装し、サブスクリプション条件とCCSPの報告要件を継続的に遵守することで、潜在的な監査ペナルティを回避する必要があります 。
 * F. 全ての契約を確認する： サービスを開始する前に、Red Hat EA（該当する地域版）、適用される製品付属文書/EULA、および特定のCCSPプログラム契約を徹底的に確認してください 。
 * G. 最終免責事項： 本レポートは情報提供を目的としたものです。理解を確認し、適用される全ての条件を確実に遵守するため、サービスを開始する前に、必ずRed Hatの営業担当者またはパートナー担当者と直接相談し、独立した法的助言を求めてください。
